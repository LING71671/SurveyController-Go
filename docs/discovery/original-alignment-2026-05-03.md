本次对齐发生在 Go 版进入 `v0.9` 运行时预览、完成 mock runner、问卷星 HTTP preview/dry-run、1000 并发本地压力脚本和脚本参考文档之后。目标是防止长会话推进时偏离原项目，同时再次确认 `v1.0` 的边界：

- `v1.0` 不做 GUI，也不把桌面依赖放进 core。
- 主线继续追求高速、高效、轻量、高性能。

## 对 Go 版 v1.0 的校准

Go 版不应该复刻原项目的桌面应用形态。当前更合理的架构是：

1. `internal/domain` 保持强类型问卷模型。
2. `internal/provider/*` 负责平台解析、检测和平台私有字段保留。
3. `internal/answer*` 负责答案策略纯函数和 answer plan。
4. `internal/runner`、`internal/engine` 负责并发、状态、事件和执行边界。
5. `cmd/surveyctl` 只暴露命令行入口，不承载业务逻辑。

这条路线和原项目的能力目标一致，但实现形态更轻。原项目的 GUI、打包器、更新器、AI 服务和真实浏览器交互都不能提前压进 `v1.0`，否则会损害当前最重要的并发、内存和可测试性目标。

## 下一批适合迁移的能力

### 1. 本地链接和 QR 文本提取

原项目支持二维码和链接输入。Go 版 `v1.0` 可以先做本地文本层能力：

- 从用户粘贴文本、已解码二维码文本或文件中提取问卷 URL。
- 调用 provider matcher 判断问卷星、腾讯问卷、Credamo。
- 不做图片解码，不访问网络，不提交数据。

这能让 CLI 更接近真实使用入口，同时保持轻量。

### 2. 三平台配置生成闭环

原项目 GUI 会把页面解析结果转成可编辑配置。Go 版应继续强化 CLI：

- provider parser 输出统一 `SurveyDefinition`。
- `surveyctl config generate` 从 fixture 或 URL 预览生成 YAML。
- 生成结果只表达配置，不混入运行状态。

### 3. 答案策略纯函数补齐

原项目包含随机姓名、手机号、身份证号、随机文本、信效度和反向填充。Go 版应优先迁移可测试的纯函数：

- 随机文本和格式化文本生成。
- 多选数量约束、必填题和附加填空。
- 信效度、维度一致性和反向题 schema。

这些能力应在提交路径之前完成，避免运行时临时拼策略。

### 4. HTTP-first 运行预览

1000 并发只是起点。Go 版应继续把 HTTP dry-run、事件流、预算检查和 CI smoke 做扎实：

- 预编译 answer plan。
- 减少热路径分配。
- worker pool、transport、event sink 都有明确资源上限。
- benchmark 和 stress 脚本覆盖常见规模。

### 5. 浏览器兼容路径

浏览器路径仍有价值，但应该是兼容 fallback，不是高并发主线：

- browser doctor/probe 保持独立超时。
- browser parser 和 runtime 要通过小接口隔离。
- 不让 Playwright 依赖污染 core 的纯函数包。

## 暂缓项

以下内容不进入 `v1.0`：

- GUI、托盘、主题和桌面打包。
- 自动更新器和联系方式展示。
- 图片二维码解码。
- AI 主观题服务。
- 随机 IP/地区在线服务。
- 绕过登录、验证、风控或平台限制的能力。

这些能力可以在 `v1.1+` 作为薄外壳或可选插件讨论，前提是 core/CLI 已经稳定。
